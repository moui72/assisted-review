## constitution.md

- [x] **[S]** Principle I ("Local-Only Today; Hosted Multi-User Is a Planned Future Direction") is written as pure description of current architecture, not a binding constraint — it explicitly says the future design pass is "out of scope for incremental patches," which reads as permission to defer *any* hardening (even things that wouldn't require the full multi-user redesign, like adding basic rate limiting or a warning if bound to a non-loopback interface) until a big-bang rewrite that has no scheduled trigger. A principle that constrains nothing today isn't really a principle.
  > `/ardd-refine constitution restate Principle I so loopback-only binding remains a hard requirement today (not merely descriptive), while keeping the note that auth/multi-tenancy design is deferred to a dedicated future pass`
  Resolved differently than the command suggested: retitled Principle I back to "Local-Only, No Off-Machine Exposure" as a hard, testable constraint (v2.0.0 → v3.0.0), and moved the forward-compatibility guidance to `CLAUDE.md` as a standing code-writing guideline instead of constitutional scope language. `api.md`/`infrastructure.md` updated to match.

- [x] **[Q]** Known gaps/shortcuts are documented three different ways across artifacts: `infrastructure.md` has a formal `## Production Annotations` section, `adapters.md` uses inline "Confirmed gap (future work)" prose, and `ui.md` uses inline "Confirmed gap (not intentional)" prose inside its States list. `/ardd-plan`'s "Production Annotation Summary" step presumably needs to harvest these reliably — should every artifact use the same formal section, or is inline-per-context intentional and the plan step is expected to read prose instead?
  Resolved: standardized on `## Production Annotations` (infrastructure.md's format) everywhere. Converted `adapters.md` and `ui.md`'s inline notes, and codified the requirement in `constitution.md`'s Development Workflow section (v3.0.0 → v3.1.0).

## infrastructure.md

- [x] **[Q]** `infrastructure.md` (transport/auth/retry mechanics) and `adapters.md` (shape mapping) both organize around the same four external sources (GitHub, GitLab, Jira, Claude) and must be updated together whenever a source's integration changes. Is the split earning its keep as two files, or would a single "Integrations" artifact with per-source subsections (transport + mapping together) reduce the chance of one half drifting out of sync with the other?
  Resolved: merged `adapters.md` into `infrastructure.md`. Each source (GitHub, GitLab, Jira, Claude) now has transport and shape-mapping together under one `### Source` subsection in Integration Components; the shared diff-parsing and adapter-wide Normalization Rules moved in too. `adapters.md` deleted; all cross-references (`api.md`, `datamodel.md`, `features.md`, `STATUS.md`) updated.

- [x] **[Q]** `parse-diff.ts` (now documented under infrastructure.md's "Unified diff → RawHunk / Chunk" subsection, formerly in `adapters.md`) is a direct TS port of an earlier Python script rather than an adopted, maintained unified-diff parsing library — a reinvented-wheel pattern per the Standardness lens. Given GitLab's REST-fallback diff reconstruction (`fetchGitLabDiffREST`) is written to this parser's exact expectations, is a swap to a library still worth evaluating, or is the hand-rolled parser a deliberate choice (e.g., to keep the GitLab reconstruction step simple) that should just be documented as such?
  Resolved: adopted the `parse-diff` npm package. `src/parse-diff.ts` is now a thin adapter mapping its `File`/`Chunk` output onto this codebase's `RawHunk[]`; `fetchGitLabDiffREST`'s reconstruction needed no changes (verified against the library's file-boundary-detection heuristic). Two edge-case behaviors changed (non-standard line inside a hunk now treated as context instead of ending the hunk; unrecognized lines outside a hunk no longer warn) — both low-impact since real `gh`/`glab` diffs never hit them; tests updated to document the new behavior rather than deleted silently. Full suite (321 tests) passes; backend coverage stays at 93.81%/94.45% stmts/lines, above the 90% gate.

## datamodel.md

- [x] **[R]** Three parallel "AI note" shapes exist — `AiNote` (in-memory, mock-only), `StoredNote` (persisted), `DisplayNote` (frontend-only union of the two) — each with its own field set that must be hand-kept in sync. `CLAUDE.md` documents this as intentional, but any future field addition to AI commentary (e.g., a new note kind, a rating field) means touching three type definitions and the mapping between them, with no compiler-enforced link.
  Resolved, in the opposite direction from the original framing: rather than accept the duplication as intentional, collapsed it. `AiNote` deleted — `Chunk.ai_notes` is now `StoredNote[]`, and `attachMockNotes` fills fake `id`/`chunk_id`/`created_at` instead of using a bespoke shape. `DisplayNote` also deleted from `AiCommentary.tsx`; the notes list is now `StoredNote[]` directly, and only the live-streaming preview (which genuinely has no id yet) keeps a narrow `NotePreview = Pick<StoredNote, 'kind'|'body'|'prompt'|'suggested_action'>` type. This removed both hand-written field-copying blocks in `App.tsx`. One new mechanism was needed: since mock notes now carry a (fake) id, the old "id present → deletable" check no longer works, so delete-ability is now gated by an explicit `deletableNoteIds` set built from `ReviewState.notes`. `CLAUDE.md`/`datamodel.md` updated; two `parse-diff`-adjacent test fixtures (`preload.test.ts`, `review.test.ts`) updated to the new required fields. Full suite (321 tests) passes; typecheck and build clean.

- [x] **[R]** `migrate()` currently backfills exactly two historical gaps (missing `version`, missing `pr.platform`) via ad hoc special-casing rather than a general versioned-migration framework. This is proportionate at `STATE_VERSION = 1`→2, but there's no established pattern for what happens at version 5 or 10 — risk of an increasingly tangled if/else chain rather than a scalable migration list.
  Resolved: refactored into an ordered `MIGRATIONS` list of two step shapes — `{ sinceVersion, apply }` (gated on stored version, bumps it) and `{ backfill }` (gated on its own field-presence check, for schema additions that shipped without a version bump, like GitLab's `pr.platform`). Adding the next gap is now pushing a new list entry rather than editing `migrate()`'s body. All 6 existing `migrate` tests pass unchanged (33/33 in `state.test.ts`); full suite, typecheck, lint all clean.

## api.md

- [x] **[R]** `POST /api/action` mutations against the single in-memory `AppContext` have no per-request serialization. Node's event loop can interleave two near-simultaneous requests' `applyAction`/`saveState` work (e.g., two `add_comment` calls both reading the same pre-mutation state before either finishes saving), producing a lost update — even though each individual `saveState()` write is itself atomic on disk. Low likelihood for a single local reviewer, but worth naming since it's a different race than the already-documented "no file locking across processes" gap.
  Resolved: added `src/mutex.ts` (a small FIFO in-process mutex) and routed both mutation sites — `POST /api/action` and the Claude SSE stream's terminal `add_note` write — through it, so overlapping requests can no longer race. `POST /api/submit` has the same class of race (read state, slow network call, write state) but was left as-is — rare/terminal/user-initiated, not rapid-fire like the other two. Proven with a deterministic `tests/mutex.test.ts` (verified it actually discriminates: manually reverted the fix and confirmed the regression test then fails) rather than trying to force real request overlap over HTTP, which turned out not to be reliably reproducible via `fetch()` (client buffers streamed bodies before sending, so it can't hold a request open server-side). `api.md` documents the fix and the explicitly-out-of-scope `/api/submit` case. Full suite (325 tests), typecheck, lint, and build all clean.

- [x] **[R]** Mac-vs-Windows keyboard hint labeling relies on `navigator.userAgent` sniffing (`IS_MAC` in `App.tsx`) — a well-known fragile pattern (spoofable, and browsers are progressively freezing UA strings). Low stakes here since it only affects cosmetic keyboard-shortcut glyphs, not behavior, but worth naming as the kind of "reinventing rather than using the standard mechanism" the Standardness lens is meant to catch.
  Documented (no code change): added to `ui.md`'s Production Annotations as an acknowledged risk, not fixed — low enough stakes that a UA-sniffing replacement isn't warranted right now.
