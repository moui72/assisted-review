# Plan: Provider-Neutral AI Commentary

## Goal

Deliver provider-neutral AI commentary with Codex support, provider-specific model selection, user-controlled stop/regenerate behavior, and a repaired drafted-against SHA stale-submit guard.

## Scope

Included:

- Replace the Claude-only AI stream path with a provider-neutral adapter boundary while preserving existing Claude behavior as the default.
- Add Codex as a local subprocess AI provider through `codex exec`, using the same prompt/context and stream contract as the Claude adapter.
- Add global AI provider/model configuration through `AiProviderConfig`, `GET /api/ai-config`, `POST /api/ai-config`, and Settings UI controls.
- Add Stop and Regenerate affordances for active AI streams and persisted initial notes.
- Incorporate feedback F001 by persisting the drafted-against SHA separately from the latest fetched SHA and using it for submit stale checks.
- Preserve the existing SSE event shape (`delta`, `done`, `error`) and keep `/api/claude` as a compatibility alias for `/api/ai`.

Not included:

- Hosted/multi-user auth, remote AI API SDK integration, or any network-exposed server behavior.
- A general retry policy for fetch/Jira/AI streams.
- Per-note provider/model provenance in `StoredNote`.
- Changes to chunk grouping, comment anchoring beyond the drafted-SHA fix, or submit UX beyond existing stale-warning behavior.

## Technical Approach

Implement the artifact decisions now recorded in `constitution.md`, `datamodel.md`, `infrastructure.md`, `api.md`, and `ui.md`.

Create a provider-neutral AI adapter layer that accepts a built prompt, provider config, optional repo-access stream options, and the existing stream handlers. Move the current Claude subprocess behavior behind that interface and add a Codex adapter that invokes `codex exec` as a local subprocess. The server's AI route should load `AiProviderConfig`, resolve the active provider, build prompts exactly once from the existing prompt builders, and dispatch through the adapter.

Add `AiProviderConfig` persistence in `STATE_DIR/ai-config.json` using the same atomic write convention as other state files. Missing config defaults to `{ provider: 'claude' }`; blank model fields are omitted so each provider CLI uses its default. The Settings UI reads and saves that config and keeps provider/model selection independent from per-repo investigation access.

Rename the frontend API helper and server route conceptually from Claude-specific to AI-specific while preserving compatibility. `/api/ai` is the canonical SSE route; `/api/claude` remains an alias that uses the configured provider. The browser stream helper returns the existing close function; Stop uses it and clears live preview without persisting partial text. Regenerate deletes the persisted initial note and starts a new empty-question stream for the same target.

For F001, add a durable drafted-against SHA field to `ReviewState` with a migration/backfill path. `loadState()` may continue refreshing the latest fetched SHA, but submit payload construction must use the drafted-against SHA for inline-comment stale checks. Tests should prove that reopening a review after the PR head advances does not overwrite the drafted-against SHA used by `submitReview()`/`submitGitLabReview()`.

## Phase Breakdown

Phase lists are plan work-items, not live checklists — progress is tracked in the linked tasks file.

### Phase 1: Artifact-Aligned Data and API Contracts

Delivers the persistent configuration and stale-SHA contract needed before provider UI or adapters can rely on it.

Depends on: artifact updates in this plan.

- Add shared `AiProviderConfig` types and storage helpers for `STATE_DIR/ai-config.json`, defaulting to Claude when absent.
- Add `GET /api/ai-config` and `POST /api/ai-config` with provider validation, model trimming, and atomic persistence.
- Add `ReviewState` drafted-against SHA support, migration/backfill behavior, and submit-path use of the drafted SHA for feedback F001. [artifacts: datamodel]

### Phase 2: Provider-Neutral Stream Boundary

Delivers a Claude-preserving adapter boundary and the new Codex provider without changing the browser's SSE event contract.

Depends on: Phase 1.

- Create the provider-neutral AI adapter interface and move current Claude subprocess streaming behind it.
- Add the Codex subprocess adapter using `codex exec`, mapped into the same `onDelta`/`onDone`/`onError`/cancel contract.
- Change the server stream route to canonical `/api/ai`, load `AiProviderConfig`, dispatch to the selected provider, pass provider-specific model arguments, and keep `/api/claude` as a compatibility alias.
- Preserve investigation-mode prompt/context behavior for both providers, with adapter-level errors when a provider cannot safely support a requested repo-access mode.

### Phase 3: Settings and Stream Controls

Delivers reviewer-facing provider/model selection plus Stop and Regenerate controls.

Depends on: Phase 2.

- Add frontend API helpers for AI config and update app state to load/save provider/model preferences.
- Add Settings controls for provider selection and the active provider's model field, keeping blank as provider default.
- Rename user-visible generic "Claude" labels to "AI" while retaining provider-specific names in Settings and error messages where useful.
- Add Stop behavior for foreground streams: close the active stream, clear live preview, and avoid persisting partial notes.
- Add Regenerate behavior for persisted initial notes: delete the note and start a fresh empty-question stream for the same chunk or overview.

### Phase 4: Verification and Compatibility

Delivers confidence that existing Claude behavior still works while Codex/model/stop/regenerate/stale-SHA paths are covered.

Depends on: Phase 3.

- Add/adjust backend tests for AI config persistence, `/api/ai` route dispatch, `/api/claude` alias behavior, provider model argument passing, provider absence errors, Codex adapter output mapping, and drafted-SHA stale checks.
- Add/adjust frontend component tests for Settings provider/model controls, Stop, Regenerate, and provider-neutral labels.
- Run lint, typecheck/build, unit tests with coverage, and the Playwright smoke test required by the constitution.
