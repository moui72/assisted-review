## datamodel.md

- [x] **[Q]** `chunk_id`s are sequential positions assigned fresh by `parseDiff()`/`groupChunks()` on every `loadReview()` call (`src/review.ts`), but `ReviewState.comments`/`flagged`/`viewed`/`notes` persist across reopens keyed by those same ids, with no reconciliation. If the same PR is reopened after upstream commits change the diff shape (hunks added/removed/reordered), old entries silently point at whatever chunk now happens to hold that id â€” not necessarily the same code â€” or at an id that no longer exists at all. The latter isn't just a display glitch: `submit.ts`'s `byId.get(c.chunk_id)` returns `undefined` for an orphaned comment and the loop does `continue`, so that comment is silently dropped from the posted review with nothing telling the reviewer it never went out.
  Resolved: accepted unstable `chunk_id`s (rejected content-derived hashing â€” discussed trade-off was that revising the exact commented-on hunk, the most common real case, would still change its hash and orphan the comment anyway, plus it complicates merged-chunk identity and requires a one-time state migration). Direction instead: detect displacement on reopen, de-anchor affected entries, warn the reviewer, and let them pick a new anchor. Logged as backlog feature `displaced-comment-reanchoring` in `features.md`; run `/ardd-plan displaced-comment-reanchoring` to design and apply across `datamodel.md`/`api.md`/`ui.md`.

- [ ] **[R]** `ReviewState.head_sha` is unconditionally overwritten to the just-fetched value on every `loadState()` call (`src/state.ts`'s `// Keep the persisted state; just refresh head_sha` comment), which only preserves useful staleness signal within a single continuous session (open once, draft, submit without reopening). If the reviewer reopens the same PR/MR before submitting, `head_sha` is refreshed to whatever's currently latest, so the submit-time staleness check (`shaOnPr`/`glabShaOnMr` against this refreshed value) can never detect that anything changed since the reviewer started drafting â€” it's comparing "latest" against "latest." Low likelihood combined with the `chunk_id` drift risk above, but worth naming since both stem from the same root cause (no version-aware reconciliation on reopen).

## api.md

- [x] **[Q]** `POST /api/submit`'s GitLab path (`submitGitLabReview`) returns `ok: true` (and the route then stamps `state.submitted`, permanently blocking further submits with `410`) even when some inline comments failed to post (`comment_errors` non-empty). Once `submitted` is set, there's no way to retry just the failed comments through this tool â€” the reviewer's only recourse is posting them manually on GitLab.
  Resolved: don't stamp `submitted` on partial success; withhold the summary-note/approve call until all inline comments post successfully; add per-request retry/backoff for transient failures; if still partial, persist per-comment success/failure in `ReviewState` so a later retry only re-attempts the failed ones (no duplicate posts). Logged as backlog feature `resilient-gitlab-submit` in `features.md`; run `/ardd-plan resilient-gitlab-submit` to design and apply across `api.md`/`datamodel.md`/`infrastructure.md`.
